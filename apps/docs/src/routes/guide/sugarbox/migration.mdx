# Save migration

Saving is easy until you ship a version, change your data shape, and users try to load something from last week.

Sugarbox has a migration system so you can keep old saves working, even when you add, rename, or restructure fields in your story state.

This page explains:

- what `saveVersion` is and how it’s used
- how compatibility checks work (`saveCompat`)
- how to register migrations during `SugarboxEngine.init()`
- what to migrate (and what not to)

## The moving parts

### `config.saveVersion`

Every save is tagged with a semantic version string like `"0.3.0"`.

When you call `saveToSaveSlot()` or `saveToExport()`, the engine stores `saveVersion` alongside the saved data. Later, when loading, Sugarbox uses that version with `saveCompat` to categorize the save as:

- compatible (`"compat"`): load directly
- old (`"old"`): attempt migration
- too new (`"new"`): throw

You should bump `saveVersion` any time you make a change that affects how saved state should be interpreted.

Examples of changes that should bump `saveVersion`:

- you rename a field in `vars`
- you change a field’s type (string → number, object → array, etc.)
- you move data around (for example `player.gold` → `wallet.gold`)
- you introduce a new required field that older saves won’t have

### `config.saveCompat`

`saveCompat` controls how the engine decides whether a save version is **compatible**, **old**, or **too new** compared to the engine’s `saveVersion`.

This is only a version-comparison policy. It does not “auto-migrate” on its own and it does not change what migrations do. It just affects which code path the engine uses when you call `loadSaveFromData()` (which is what slot loads and export loads end up using).

What the modes mean in this repo’s code:

- `"strict"` (default): saves are only compatible when their **major** and **minor** match the engine’s. If the save’s major differs, it’s treated as old/new. If the major matches but the save’s minor is lower, it’s treated as **old**. If the save’s minor is higher, it’s treated as **new**.
- `"liberal"`: saves are compatible when the **major** matches and the save’s **minor** is **less than or equal** to the engine’s. A higher minor is still treated as **new**.

Important: the current compatibility function does not use patch versions at all when deciding compat/old/new.

If a save is considered **new**, the engine throws. If it’s considered **old**, the engine will attempt to migrate it (see below), and if it’s **compat**, it loads directly.

### Migrations

A migration is a small function that transforms **state** from one `saveVersion` to the next.

Sugarbox expects migrations at init time in the `migrations` array:

- each entry has a `from` version (the version you are migrating from)
- and `data` which includes:
  - `to`: the version after this migration completes
  - `migrater`: a function that takes the current state (for that save version) and returns the migrated state

How migrations are actually applied in code:

- Migrations only run when a save is classified as **old** by the compatibility check.
- The engine loops from the save’s `saveVersion` toward the engine’s `saveVersion`:
  - it looks up a migrater registered for the *current* version string
  - runs it to produce a new state
  - updates the “current save version” to the migrater’s `to`
  - repeats until it reaches the engine version
- If any step is missing (no migrater for the current version), loading fails with an error.
- If any migrater throws, the engine restores the original in-memory state and rethrows.

So yes, migrations are chained, but only if you provide every hop the engine needs from the save’s version to the engine’s version.

## A practical workflow

### 1) Decide what you version

In Sugarbox, the most common thing you version is your story variables (`vars`) and any save metadata you rely on.

Settings and achievements are persisted separately from save slots, so versioning your *save* schema usually focuses on variables and story progress.

### 2) Bump `saveVersion` when you change schema

When you make a schema change, bump the version you pass into init:

```ts
import { SugarboxEngine } from "sugarbox";

const engine = await SugarboxEngine.init({
  name: "MyStory",
  startPassage: { name: "Start", data: "..." },
  otherPassages: [],
  vars: { player: { gold: 10 } },
  config: {
    saveVersion: "0.2.0",
    saveCompat: "strict",
  },
});
```

### 3) Add a migration from the previous version

Say version `0.1.0` stored `gold` at the top level, and `0.2.0` moves it under `player.gold`.

Add a migration:

```ts
import { SugarboxEngine } from "sugarbox";

const engine = await SugarboxEngine.init({
  name: "MyStory",
  startPassage: { name: "Start", data: "..." },
  otherPassages: [],
  vars: { player: { gold: 10 } },
  config: {
    saveVersion: "0.2.0",
    saveCompat: "strict",
  },
  migrations: [
    {
      from: "0.1.0",
      data: {
        to: "0.2.0",
        migrater: (oldSave) => {
          // `oldSave` is the previous save structure for that version.
          // Keep this function defensive: old data can be weird.
          const gold =
            typeof oldSave.gold === "number" ? oldSave.gold : 0;

          return {
            ...oldSave,
            gold: undefined,
            player: {
              ...(typeof oldSave.player === "object" && oldSave.player ? oldSave.player : {}),
              gold,
            },
          };
        },
      },
    },
  ],
});
```

A few notes about the example:

- Keep migrations purely about data shaping. Avoid calling into UI, network, or game logic.
- Be defensive about types. Old saves can come from older builds, forks, or corrupted storage.
- Prefer explicit defaults over letting `undefined` leak into “required” fields.

## Chaining migrations cleanly

It’s tempting to write one big migrater from `0.1.0` straight to `1.0.0`. Don’t.

Small steps are easier to reason about and easier to test. Chain them version-by-version.

Example:

```ts
const migrations = [
  {
    from: "0.1.0",
    data: {
      to: "0.2.0",
      migrater: (save) => ({ ...save, addedIn020: true }),
    },
  },
  {
    from: "0.2.0",
    data: {
      to: "0.3.0",
      migrater: (save) => ({ ...save, renamedIn030: save.addedIn020 }),
    },
  },
  {
    from: "0.3.0",
    data: {
      to: "0.4.0",
      migrater: (save) => ({ ...save, finalShape: { ok: true } }),
    },
  },
];
```

With this approach, you can remove old migration steps later if you intentionally decide to drop support for very old saves.

## What should a migration touch?

A safe rule: migrations should only touch persisted data structures.

Most commonly:

- `vars` fields you moved/renamed
- data format changes (string → number, array → map, etc.)
- adding missing fields with defaults
- normalizing inconsistent data

Avoid:

- writing migrations that depend on passage content
- trying to re-run story logic during migration
- doing anything that needs external data at load time

If you truly need to recompute derived data, it’s usually better to compute it lazily during gameplay rather than persisting it.

## Compatibility: strict vs liberal

### When to use `"strict"`

Use `"strict"` when you want version bumps to be explicit:

- You want saves from a different **minor** version to be treated as **old/new**, not silently accepted.
- You want to force yourself to write migrations when you make minor-version schema changes.

In this mode, if you bump `saveVersion`’s minor, older saves will be categorized as `"old"` and require migrations. Newer saves (higher minor) will be rejected as `"new"`.

### When to use `"liberal"`

Use `"liberal"` when you want backward-compatibility across lower minor versions (within the same major):

- Saves with the same major and a lower minor will be treated as `"compat"` and loaded without migration.
- Saves with a higher minor are still rejected as `"new"`.

This can be handy during development, but it also means you can accidentally load older data shapes without migrating them, and then run into runtime assumptions later.

## Migration events

Migration also emits lifecycle events that are useful for UI:

- `:migrationStart`
- `:migrationEnd`

You can use this to show a loading screen or log errors centrally.

Note: the engine in this repo exposes a typed `on()`/`off()` API for events. If you prefer the DOM-style `addEventListener`, you’ll want to follow whatever event wiring you use elsewhere in your app. The important part is the payload.

```ts
engine.on(":migrationStart", (event) => {
  console.log("migrating", event.detail.fromVersion, "->", event.detail.toVersion);
});

engine.on(":migrationEnd", (event) => {
  if (event.detail.type === "success") {
    console.log("migration complete");
  } else {
    console.error("migration failed", event.detail.error);
  }
});
```

## Testing migrations (worth doing)

A migration bug is a player progress loss bug.

At minimum:

- keep a fixture of representative save data for each version
- run the migrater functions in tests
- assert the new shape, especially required fields and invariants

If your story depends on specific fields always existing, assert that in tests, not just in type definitions.

## Common patterns

### Rename a field

```ts
migrater: (save) => {
  const { oldField, ...rest } = save;
  return { ...rest, newField: oldField ?? null };
}
```

### Split one field into two

```ts
migrater: (save) => {
  const fullName = typeof save.fullName === "string" ? save.fullName : "";
  const [firstName, ...rest] = fullName.trim().split(/\s+/);
  const lastName = rest.join(" ");

  return { ...save, firstName, lastName };
}
```

### Add a required field with a default

```ts
migrater: (save) => ({
  ...save,
  difficulty: save.difficulty ?? "normal",
})
```

## A note about typing

Migrations are written against old shapes, and old shapes are, by definition, not your current TypeScript types.

You have a couple options:

- keep migrations loosely typed internally and validate as you go
- define explicit `SaveVx_y_z` types for old versions if the project is stable and you value safety

Either way, the goal is the same: take whatever is in storage and turn it into something your current engine can load confidently.
