---
title: Events
---

Sugarbox is headless, so events are how you wire the engine to your UI, persistence, and side effects.

Under the hood, each `SugarboxEngine` instance exposes a small typed event API:

- `engine.on(eventName, listener)`
- `engine.off(eventName, listener)`

`on()` returns an unsubscribe function, so you often don’t need to call `off()` yourself unless you prefer that style.

Sugarbox emits a set of custom events whose names start with `:`. The payload lives on `event.detail`.

## Quick example

```ts
import { SugarboxEngine } from "sugarbox";

const engine = await SugarboxEngine.init({
  name: "Example",
  startPassage: { name: "Start", data: "Hello" },
  otherPassages: [],
  vars: { gold: 10 },
});

const offPassage = engine.on(":passageChange", (event) => {
  const { oldPassage, newPassage } = event.detail;
  console.log("passage:", oldPassage?.name, "->", newPassage?.name);
});

const offState = engine.on(":stateChange", (event) => {
  const { oldState, newState } = event.detail;
  console.log("gold:", oldState.gold, "->", newState.gold);
});

engine.navigateTo("Start");
engine.setVars((s) => {
  s.gold += 5;
});

// later, when you want to stop listening:
offPassage();
offState();
```

## Event list

These are the engine-emitted events you can depend on.

### `:passageChange`

Fires when you navigate to a different passage.

Payload:

- `oldPassage`: previous passage object, or `null` if not available
- `newPassage`: new passage object, or `null` if not available

Common uses:

- update your router or render target
- run passage-level analytics
- trigger music, ambience, or one-shot effects

### `:stateChange`

Fires when `setVars()` changes story state.

Payload:

- `oldState`: state before the change
- `newState`: state after the change

Notes:

- If you’re changing state inside a `:stateChange` listener, be careful with loops. The engine exposes an `emitEvent` flag on `setVars()` so you can avoid re-entrancy when you know what you’re doing.
- Depending on config, the engine may or may not clone the previous state when emitting. If you need to compare deeply for UI updates, treat both as read-only snapshots.

### `:achievementChange`

Fires when achievements data changes.

Payload:

- `old`: previous achievements object
- `new`: current achievements object

Achievements are meant to live across saves (they are not “slot data” in the same way story variables are).

### `:settingChange`

Fires when settings data changes.

Payload:

- `old`: previous settings object
- `new`: current settings object

Settings are also stored separately from save slots. Typical examples are audio volume, font size, theme, accessibility toggles.

### Save and load lifecycle events

These make it easy to show UI like “Saving…” and to capture errors in one place, without wrapping every call site.

#### `:saveStart`

Fires when a save operation begins.

Payload:

- `slot`: `"autosave" | "export" | "recent" | number`

#### `:saveEnd`

Fires when a save operation completes.

Payload:

- Success: `{ type: "success", slot }`
- Error: `{ type: "error", slot, error }`

#### `:loadStart`

Fires when a load operation begins.

Payload:

- `slot`: `"autosave" | "export" | "recent" | number`

#### `:loadEnd`

Fires when a load operation completes.

Payload:

- Success: `{ type: "success", slot }`
- Error: `{ type: "error", slot, error }`

#### `:deleteStart`

Fires when a delete begins.

Payload:

- `slot`: `"autosave" | number`

#### `:deleteEnd`

Fires when delete completes.

Payload:

- Success: `{ type: "success", slot }`
- Error: `{ type: "error", slot, error }`

### Migration lifecycle events

If your engine version changes and you use save migrations, you can surface progress and failures cleanly.

#### `:migrationStart`

Payload:

- `fromVersion`: semantic version string
- `toVersion`: semantic version string

#### `:migrationEnd`

Payload:

- Success: `{ type: "success", fromVersion, toVersion }`
- Error: `{ type: "error", fromVersion, toVersion, error }`

## Practical patterns

### 1) Unsubscribe cleanly

If you’re wiring this into a UI framework, unsubscription matters. Keep stable listener functions and remove them when a component unmounts.

```ts
const off = engine.on(":stateChange", (event) => {
  const { oldState, newState } = event.detail;
  console.log(oldState, newState);
});

// later...
off();
```

### 2) Show save UI without coupling to call sites

```ts
const offSaveStart = engine.on(":saveStart", (event) => {
  const { slot } = event.detail;
  console.log("saving...", slot);
});

const offSaveEnd = engine.on(":saveEnd", (event) => {
  if (event.detail.type === "success") {
    console.log("save done:", event.detail.slot);
  } else {
    console.error("save failed:", event.detail.slot, event.detail.error);
  }
});

// later...
offSaveStart();
offSaveEnd();
```

### 3) Avoid feedback loops

If you update state in response to state changes, you can accidentally create an infinite loop. The simplest fix is to set up your listeners so they do not call `setVars()` based on `:stateChange`, or to explicitly disable emission when you know you’re doing a derived write.

```ts
engine.on(":stateChange", (event) => {
  const { newState } = event.detail;

  // Example: keep a derived field in sync, but do not emit a second event.
  if (typeof newState.level === "number") {
    engine.setVars((s) => {
      s.levelLabel = `Level ${s.level}`;
    }, false);
  }
});
```

## Notes on TypeScript

Sugarbox’s events are custom events, so in vanilla DOM typing they show up as `Event`. In most projects you’ll either:

- cast to `CustomEvent` in the handful of places you need `detail`, or
- wrap event registration in a tiny helper that narrows the types for you.

The important bit is consistent: the payload is always in `event.detail`.
