---
title: Saving and Loading Saves
---

Sugarbox can persist story progress to storage, export saves as a string, and load them back later. Saves are versioned and can be migrated when your data shape changes.

This page covers the practical bits: setting up persistence, working with save slots, exporting/importing, and deleting save data.

## What “saving” means in Sugarbox

A save generally includes:

- your story variables (`engine.vars`)
- metadata the engine needs to reconstruct state history
- the current passage id
- PRNG seed metadata (stored inside the state snapshots) so outcomes can be reproducible across load

Sugarbox also stores two other buckets separately from save slots:

- **settings** (volume, theme, etc.)
- **achievements**

Those are meant to persist outside of save slots. A save slot is “this run”, while settings and achievements are “this device / profile”.

## Persistence: required for save slots, achievements, and settings

To use save slots (autosave or numbered slots), you need a persistence adapter in `config.persistence`.

The engine uses the persistence adapter for:

- save slots (`saveToSaveSlot`, `loadFromSaveSlot`, `getSaves`, deletes)
- achievements storage (`setAchievements` persists)
- settings storage (`setSettings` persists)
- loading achievements/settings during `init()`

### The actual persistence adapter shape

In the library code, the persistence adapter is typed as:

- `set(key, data) -> Promise<unknown>`
- `get(key) -> Promise<string | undefined | null>`
- `delete(key) -> Promise<unknown>`
- `keys?() -> Promise<Iterable<key>>` (optional)

A couple important details:

- `keys()` is optional, and it returns an `Iterable`, not necessarily an array.
- `get()` may return `null` or `undefined` when a key doesn’t exist.

### Example: localStorage adapter

```ts
type PersistenceAdapter = {
  get: (key: string) => Promise<string | undefined | null>;
  set: (key: string, value: string) => Promise<unknown>;
  delete: (key: string) => Promise<unknown>;
  keys?: () => Promise<Iterable<string>>;
};

export function createLocalStorageAdapter(prefix = "sugarbox:"): PersistenceAdapter {
  return {
    async get(key) {
      // localStorage returns `string | null`, which fits the engine’s expectations
      return localStorage.getItem(prefix + key);
    },
    async set(key, value) {
      localStorage.setItem(prefix + key, value);
    },
    async delete(key) {
      localStorage.removeItem(prefix + key);
    },
    async keys() {
      const out: string[] = [];
      for (let i = 0; i < localStorage.length; i++) {
        const fullKey = localStorage.key(i);
        if (!fullKey) continue;
        if (!fullKey.startsWith(prefix)) continue;
        out.push(fullKey.slice(prefix.length));
      }
      return out;
    },
  };
}
```

Then attach it during initialization:

```ts
import { SugarboxEngine } from "sugarbox";
import { createLocalStorageAdapter } from "./local-storage-adapter";

const engine = await SugarboxEngine.init({
  name: "MyStory",
  startPassage: { name: "Start", data: "..." },
  otherPassages: [],
  vars: { gold: 0 },

  config: {
    persistence: createLocalStorageAdapter(),
    saveSlots: 10,
  },
});
```

Notes:

- The engine name matters. Engines created with the same `name` share persisted data (saves, achievements, settings). Use a unique name per story.
- If you do not provide persistence, save-slot operations (save/load/delete/list) will throw. Also, achievements and settings cannot be loaded/saved.

## Save slots

Sugarbox supports:

- an **autosave** slot (no number)
- **numbered slots** (0..`saveSlots - 1`)

The engine also exposes a “recent save” concept when loading on start via `loadRecentSave()`.

### Configuring slot count

`config.saveSlots` controls how many numbered slots exist.

```ts
const engine = await SugarboxEngine.init({
  name: "MyStory",
  startPassage: { name: "Start", data: "..." },
  otherPassages: [],
  vars: {},
  config: { persistence, saveSlots: 5 },
});
```

Slot indices are zero-based. With `saveSlots: 5`, valid numbered slots are `0..4`.

## Autosave

Autosave is controlled by `config.autoSave`.

The actual config type is:

- `false` (default): no autosave
- `"passage"`: autosave when you navigate between passages
- `"state"`: autosave when you change state via `setVars()`

```ts
const engine = await SugarboxEngine.init({
  name: "MyStory",
  startPassage: { name: "Start", data: "..." },
  otherPassages: [],
  vars: {},
  config: { persistence, autoSave: "passage" },
});
```

If you want more control, keep `autoSave: false` and call `saveToSaveSlot()` explicitly from your app at the times that make sense (after choices, on checkpoints, before risky scenes, etc.).

## Listing existing saves

You can enumerate saves with `engine.getSaves()`, which is an async generator.

It yields entries like:

- `{ type: "autosave", data: ... }`
- `{ type: "normal", slot: number, data: ... }`

```ts
for await (const save of engine.getSaves()) {
  if (save.type === "autosave") {
    console.log("autosave exists", save.data);
  } else {
    console.log("slot", save.slot, "exists", save.data);
  }
}
```

This is helpful for building a save/load menu.

## Deleting saves

### Delete one slot

Use `engine.deleteSaveSlot()`:

- with no argument it deletes autosave
- with a number it deletes that numbered slot

```ts
await engine.deleteSaveSlot();   // autosave
await engine.deleteSaveSlot(2);  // slot 2
```

### Delete everything

If you want to clear all slots, use `engine.deleteAllSaveSlots()`:

```ts
await engine.deleteAllSaveSlots();
```

This iterates through any saves the engine can find and deletes them.

## Exporting and importing saves

Export is meant for share links, manual backups, support requests, or moving progress between devices.

### Export

Sugarbox can export a save bundle as a string via `saveToExport()`. Depending on config, that string may be compressed.

The engine emits save/load events during export too, so you can show UI feedback.

A typical flow:

1. Export to get a string
2. Store it in a file, clipboard, or cloud
3. Later, load that export string back

```ts
const exported = await engine.saveToExport();

// Save `exported` somewhere safe, like a file or clipboard.
// Later, on another device or after reinstall:

await engine.loadFromExport(exported);
```

## Compression

Compression is controlled by `config.compress`.

- When `compress: true`, save strings (including export data) may be compressed before being written to storage or returned for export.
- When `compress: false`, you will generally see readable JSON strings.

Note: the library only compresses when the stringified data is large enough (it avoids compressing tiny payloads).

```ts
const engine = await SugarboxEngine.init({
  name: "MyStory",
  startPassage: { name: "Start", data: "..." },
  otherPassages: [],
  vars: {},
  config: {
    persistence,
    compress: true,
  },
});
```

Compression is worth enabling if you store large variable trees, long text, or lots of history.

## Save and load events

Saves, loads, deletes, and migrations emit lifecycle events. These are useful for UI and error handling.

Relevant events for saving/loading:

- `:saveStart`, `:saveEnd`
- `:loadStart`, `:loadEnd`
- `:deleteStart`, `:deleteEnd`

Example (using the engine’s typed event API):

```ts
engine.on(":saveStart", (event) => {
  console.log("saving...", event.detail.slot);
});

engine.on(":saveEnd", (event) => {
  if (event.detail.type === "success") {
    console.log("saved", event.detail.slot);
  } else {
    console.error("save failed", event.detail.slot, event.detail.error);
  }
});
```

If you want the full list and payload shapes, see the [Events](../event) page.

## Common gotchas

### “Saving throws an error”

Make sure you provided `config.persistence`. Slot-based saving and deletion require it.

### “I changed my state shape and old saves crash”

Use save migrations and bump `config.saveVersion`. See [Save Migration](../migration).

### “My exports are huge”

Turn on `config.compress`. Also consider what you store in `vars`. Things like full UI state or derived data usually don’t belong there.

### “I need to store custom classes”

Register class constructors via `classes` in `init()` so serialization/deserialization can recreate them. See the TypeScript guide.
